## Intencion funcional
SARA debe poder gestionar tareas simples desde Chatwoot.

Ejemplos:

```text
tarea: llamar al contador
crear tarea: revisar facturas
tengo que ordenar la bandeja de entrada
que tareas tengo
listar tareas
marcar tarea 1 como hecha
completar tarea 2
complete llamar al contador
```

## Arquitectura esperada

### 1. DB
Crear migracion nueva solo para objetos `sara_`.

Tabla autorizada:

```text
sara_tasks
```

Campos minimos:
- `id uuid primary key`
- `schema_version text not null`
- `title text not null`
- `description text`
- `status text not null`
- `source text not null`
- `area_id uuid`
- `due_at timestamptz`
- `completed_at timestamptz`
- `trace_id uuid`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Estados permitidos:
- `pending`
- `completed`

Eventos en `sara_events`:
- `task_created`
- `task_completed`

Recomendado:
- usar RPCs `sara_create_task` y `sara_complete_task`, o writer canonico equivalente.
- RLS habilitado.
- acceso anon/authenticated revocado.
- ejecucion solo por `service_role`.


Responsabilidades:
- `tasksModule`: reglas de negocio y validaciones.

- `tasksModule` no conoce Chatwoot.
- `tasksStore` no clasifica intenciones.

### 4. Acciones

`tasks.create`:
- crea una tarea pendiente.
- title no vacio.
- source = `chatwoot`.
- confirma solo con `taskId` y `eventId`.

`tasks.list`:
- lista tareas `pending` por default.
- orden sugerido: `created_at desc`.
- limite default 5, maximo 10.
- read-only.

`tasks.complete`:
- completa una tarea pendiente.
- acepta `taskId` si esta disponible.
- acepta indice humano simple si la intencion lo extrae como `position`.
- `position` se resuelve contra la lista actual de pendientes con el mismo orden de `tasks.list`; no depende de memoria conversacional.
- para MVP seguro, si no hay `taskId` ni matching claro por titulo exacto/parcial, responder missingData.
- confirma solo con `taskId` y `eventId`.
