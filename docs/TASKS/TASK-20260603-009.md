## Intencion funcional
SARA debe recordar contexto conversacional efimero dentro de la conversacion actual para resolver referencias simples de forma segura.

Ejemplos esperados despues de este modulo:

```text
que tareas tengo
```

Respuesta:

```text
Estas son tus tareas pendientes:
1. llamar al contador
```

Luego:

```text
completar esa
```

Debe poder completar la tarea enfocada si el contexto tiene foco claro en una unica tarea pendiente.

Otro ejemplo:

```text
tarea: llamar al contador
```

Respuesta:

```text
Tarea creada: llamar al contador
```

Luego:

```text
completar la ultima tarea
```

Debe poder completar la tarea recien creada si sigue pendiente y el contexto no expiro.

## Arquitectura esperada

### 1. DB
Crear migracion nueva solo para objetos `sara_`.

Tabla autorizada:

```text
sara_session_contexts
```

Campos minimos segun `docs/ENTITY_CATALOG.md`:
- `id uuid primary key`
- `schema_version text not null`
- `account_id bigint not null`
- `inbox_id bigint not null`
- `conversation_id bigint not null`
- `active_module text`
- `active_flow text`
- `focused_entity_type text`
- `focused_entity_id uuid`
- `awaiting_confirmation boolean not null`
- `confirmation_payload jsonb`
- `context jsonb not null`
- `expires_at timestamptz`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Restricciones:
- maximo un contexto activo por `account_id`, `inbox_id`, `conversation_id`.
- RLS habilitado.
- anon/authenticated sin acceso.
- service_role autorizado.

Eventos en `sara_events`:
- `session_context_started`
- `session_context_updated`
- `session_context_cleared`

No implementar todavia:
- `confirmation_requested`
- `confirmation_resolved`

Esos eventos quedan reservados para una task posterior de confirmaciones.

### 4. Integracion con worker
Actualizar `bufferProcessor` o wiring equivalente para:
1. cargar contexto por `account_id`, `inbox_id`, `conversation_id` antes de `coarseClassifier`;
2. pasar `sessionContext` a `coarseClassifier` y `moduleIntentClassifier`;
3. despues de una accion ejecutada con `stateChanges`, actualizar foco conversacional;
4. si el contexto expiro, tratarlo como inexistente.

Regla:
- Si falla cargar o actualizar contexto, no debe impedir responder la accion principal, pero debe loguearse y no inventarse contexto.

### 5. Foco conversacional
Despues de acciones ejecutadas:

`tasks.create`:
- foco: `focusedEntityType = "task"`
- `focusedEntityId = taskId`
- `activeModule = "tasks"`
- `activeFlow = "task_created"`
- `context.lastTaskTitle = title`

`tasks.list`:
- si hay una unica tarea pendiente listada, foco en esa tarea.
- si hay multiples tareas, guardar lista resumida en `context.lastTaskList` con posiciones e ids.
- no enfocar una unica tarea si hay multiples.

`tasks.complete`:
- limpiar foco si la tarea enfocada fue completada.
- activeFlow sugerido: `task_completed`.

`notes.create/list/search`:
- opcional: enfocar nota si la evidencia trae `noteId` o si list/search devuelve un unico resultado.
- no romper comportamiento existente.

TTL sugerido:
- 30 minutos desde `updated_at` o `expires_at`.

### 6. Resolucion de referencias simples
Actualizar `moduleIntentClassifier` o una capa auxiliar de resolucion para `tasks.complete`.

Frases soportadas:
- `completar esa`
- `completar la ultima`
- `completar la ultima tarea`
- `marcar esa como hecha`
- `marcar la ultima tarea como hecha`

Solo resolver si:
- `sessionContext.focusedEntityType = "task"` y hay `focusedEntityId`; o
- `context.lastTaskList` tiene exactamente una tarea; o
- la frase referencia una posicion y `context.lastTaskList` contiene esa posicion.

Si hay ambiguedad:
- `missingData = ["task"]`
- no ejecutar.

No usar LLM para resolver referencias.
