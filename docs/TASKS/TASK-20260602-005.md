## Intencion funcional
SARA debe poder recibir una intencion de crear nota y dejarla preparada para ejecutarse como accion real.

Ejemplos de mensajes que deben clasificarse como `notes.create`:
- `nota: recordar que dormir poco me baja mucho el foco`
- `guarda una nota: revisar el protocolo de Barberox`
- `anota esto como idea: crear una rutina semanal de revision`

En esta task no se exige conectar el flujo completo del webhook al pipeline definitivo. El bootstrap actual puede seguir respondiendo directo como fallback. El objetivo es dejar el modulo de dominio listo, probado y trazable para conectarlo en una task posterior.

## Reglas obligatorias
- El LLM clasifica o redacta; no decide ejecutar.
- Las reglas de negocio viven en codigo.

- Toda migracion debe vivir en `db/migrations`.
- Toda DB, funcion, indice o politica nueva debe comenzar con `sara_`.

## Migracion autorizada
Crear una migracion versionada en `db/migrations`.

Objetos autorizados exclusivamente:
- `sara_events`
- `sara_notes`
- indices necesarios con prefijo `sara_`
- funcion RPC `sara_create_note`

### `sara_events`
Campos minimos:
- `id uuid primary key default gen_random_uuid()`
- `schema_version text not null`
- `event_type text not null`
- `entity_type text not null`
- `entity_id uuid`
- `trace_id uuid`
- `source text not null`
- `payload jsonb not null default '{}'::jsonb`
- `created_at timestamptz not null default now()`

### `sara_notes`
Campos minimos:
- `id uuid primary key default gen_random_uuid()`
- `schema_version text not null`
- `note_type text not null`
- `content text not null`
- `source text not null`
- `area_id uuid`
- `related_entity_type text`
- `related_entity_id uuid`
- `tags jsonb not null default '[]'::jsonb`
- `trace_id uuid`
- `created_at timestamptz not null default now()`

### Restricciones minimas
- `content` no puede ser vacio.
- `note_type` solo puede ser: `aprendizaje`, `idea`, `problema`, `riesgo`, `mejora`, `observacion`.
- Si `related_entity_id` existe, `related_entity_type` debe existir.
- RLS habilitado.
- Revocar acceso a `anon` y `authenticated`.
- Conceder ejecucion de `sara_create_note` a `service_role`.

### RPC `sara_create_note`
Responsabilidad:
- validar datos defensivamente;
- insertar en `sara_notes`;
- insertar evento `note_created` en `sara_events`;
- devolver JSON con `note_id`, `event_id`, `trace_id`, `schema_version`.

Firma sugerida:

```sql
sara_create_note(
  p_trace_id uuid,
  p_content text,
  p_note_type text default 'observacion',
  p_source text default 'chatwoot',
  p_area_id uuid default null,
  p_related_entity_type text default null,
  p_related_entity_id uuid default null,
  p_tags jsonb default '[]'::jsonb
)
```

## Comportamiento esperado

### Clasificacion fina
Si el modulo grueso es `notes` y el mensaje contiene una orden explicita de guardar/anotar/crear nota:
- `action`: `create`
- `confidence`: al menos `0.75` si hay contenido claro
- `entities.content`: contenido limpio de la nota
- `entities.noteType`: inferido si es claro; si no, `observacion`
- `missingData`: vacio si hay contenido
- `requiresConfirmation`: `false`

Si no hay contenido suficiente:
- `action`: `create`
- `confidence`: menor a `0.75` o `missingData: ["content"]`
- no se ejecuta accion

### Ejecucion
El executor solo puede llamar al modulo notes si:
- router marca `executable: true`;
- `intentConfidenceSufficient(intent)` es `true`;
- `requiresConfirmation` es `false`;
- `entities.content` es string no vacio.

Si la RPC retorna `note_id` y `event_id`:
- `ActionExecutionResult.status`: `executed`
- `evidence.noteId`: id real
- `evidence.eventId`: id real
- `stateChanges`: incluye evento `note_created`

Si falla:
- `status`: `failed`
- `error`: mensaje sanitizado
- no confirmar creacion.
